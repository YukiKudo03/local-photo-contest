# チュートリアルモード要件定義書 v2.0
## 桜井政博氏の設計哲学に基づく再定義

---

## 1. 設計原則

### 1.1 基本理念

> **「説明を読ませるな、体験させろ」**

本チュートリアルシステムは、桜井政博氏のゲームデザイン哲学に基づき、以下の原則を最優先とします。

### 1.2 5つの設計原則

| # | 原則 | 説明 | 検証基準 |
|---|------|------|---------|
| 1 | **まず体験させる** | 説明より先に操作させる | 最初のアクションまで10秒以内 |
| 2 | **待たせない** | 操作できない時間を最小化 | 各ステップ5秒で完了可能 |
| 3 | **失敗させない** | 安全に試行錯誤できる | 元に戻せる・やり直せる |
| 4 | **すぐ褒める** | 成功を即座にフィードバック | アクション後0.3秒以内 |
| 5 | **初心者と上級者の同居** | シンプルな基本と深いオプション | 2層構造で提供 |

## 2. コンテンツ制約

### 2.1 テキスト制限

| 要素 | 最大文字数 | 理由 |
|------|-----------|------|
| ステップタイトル | **15文字** | 一目で把握できる |
| ステップ説明文 | **40文字** | 3秒で読める |
| ヒントテキスト | **25文字** | 視線を奪わない |
| エラーメッセージ | **30文字** | 即座に理解できる |
| 成功メッセージ | **20文字** | 達成感を邪魔しない |

### 2.2 時間制限

| 要素 | 最大時間 | 理由 |
|------|---------|------|
| ウェルカムモーダル表示 | **3秒**で閉じられる | 離脱防止 |
| 各ステップの滞在 | **10秒**以内に次へ | テンポ維持 |
| アニメーション | **0.3秒**以内 | 待たせない |
| フィードバック表示 | **0.2秒**以内に開始 | 即時性 |
| チュートリアル全体 | **90秒**以内で完了可能 | 集中力維持 |

### 2.3 ステップ数制限

| チュートリアル種別 | 最大ステップ数 | 理由 |
|------------------|--------------|------|
| オンボーディング | **3ステップ** | 最短で価値を体験 |
| 機能別チュートリアル | **4ステップ** | 1機能を習得 |
| 高度な機能 | **5ステップ** | 例外的に許容 |

## 3. ユーザーフロー要件

### 3.1 参加者（Participant）

#### 3.1.1 最短価値到達パス

```
目標: 「作品を見て投票する」体験を最速で提供

[ログイン完了]
    ↓ 即座に（0秒）
[コンテスト一覧表示]
    ↓ ハイライト + 1行説明（3秒）
    「気になるコンテストをタップ」
    ↓
[作品一覧表示]
    ↓ ハイライト + 1行説明（3秒）
    「素敵な作品にハートを」
    ↓
[投票完了]
    ↓ 即座に
[成功フィードバック]
    「初投票おめでとう！」+ アニメーション

合計所要時間: 30秒以内
```

#### 3.1.2 オンボーディング要件

| ステップ | アクション | 説明テキスト | 成功条件 |
|---------|----------|-------------|---------|
| 1 | 作品をタップ | 「気になる写真をタップ」 | 作品詳細を開く |
| 2 | ハートをタップ | 「素敵な作品にハートを」 | 投票完了 |
| 3 | 完了 | 「準備完了！楽しんで」 | - |

**省略した従来の項目:**
- ❌ プラットフォーム紹介 → 使いながら理解
- ❌ コンテスト一覧の見方 → 直感的UIで不要
- ❌ マイページ説明 → 必要時にコンテキストヘルプ

#### 3.1.3 写真投稿チュートリアル

| ステップ | アクション | 説明テキスト | 補助機能 |
|---------|----------|-------------|---------|
| 1 | 写真を選択 | 「写真をドロップまたは選択」 | ドラッグ&ドロップ対応 |
| 2 | タイトル入力 | 「タイトルをつけましょう」 | プレースホルダーで例示 |
| 3 | 投稿ボタン | 「これで投稿完了！」 | プレビュー表示 |

**省略した従来の項目:**
- ❌ 説明文の入力ガイド → 任意項目は説明不要
- ❌ 位置情報の登録方法 → 自動取得 + 任意編集
- ❌ 撮影スポット選択 → 必要な場合のみ表示

## 7. 技術要件

### 7.1 データモデル更新

```ruby
# TutorialStep の制約追加
class TutorialStep < ApplicationRecord
  # 桜井流制約
  MAX_TITLE_LENGTH = 15
  MAX_DESCRIPTION_LENGTH = 40
  MAX_STEPS_PER_TUTORIAL = 5

  validates :title, length: { maximum: MAX_TITLE_LENGTH }
  validates :description, length: { maximum: MAX_DESCRIPTION_LENGTH }

  # 動詞で始まることを推奨（警告のみ）
  validate :title_should_start_with_verb

  private

  def title_should_start_with_verb
    verbs = %w[タップ クリック 選択 入力 確認 見て 押して 開いて]
    unless verbs.any? { |v| title&.start_with?(v) }
      Rails.logger.warn "[Tutorial] Title should start with verb: #{title}"
    end
  end
end
```

## 8. 測定要件

### 8.1 必須KPI

| 指標 | 目標値 | 計測方法 |
|------|--------|---------|
| **完了率** | 85%以上 | 開始数に対する完了数 |

| ヘルプボタン押下率 | 30%超 | UIの改善 |
| 同一ステップ繰り返し | 3回超 | 操作の明確化 |

## 10. 既存要件との差分

### 10.1 削減した項目

| 従来の項目 | 削減理由 |
|-----------|---------|
| 8ステップの運営者オンボーディング | 3ステップに集約、他は段階的開放 |
| 5ステップの参加者オンボーディング | 3ステップに集約、直感的UIで代替 |
| 詳細な機能説明 | コンテキストヘルプに移行 |
| プラットフォーム紹介 | 使いながら理解する設計に |

### 10.2 強化した項目

| 項目 | 強化内容 |
|------|---------|
| フィードバック | 即時性を重視、アニメーション追加 |
| スキップ機能 | 確認なし、いつでも再開可能 |
| テキスト制限 | 厳密な文字数制限を導入 |
| 時間制限 | 各要素に明確な時間目標 |

### 10.3 新規追加項目

| 項目 | 内容 |
|------|-----|
| 段階的機能開放 | 使い込むほど機能が解放される |
| マイルストーンフィードバック | バッジ・称号の付与 |
| A/Bテスト基盤 | 継続的な改善のための基盤 |

## 付録

### A. チェックリスト

新しいチュートリアルステップを追加する際のチェックリスト：

- [ ] タイトルは15文字以内か
- [ ] タイトルは動詞で始まっているか
- [ ] 説明文は40文字以内か
- [ ] 1ステップ＝1アクションか
- [ ] 成功時のフィードバックがあるか
- [ ] スキップ可能か
- [ ] モバイルで動作確認したか

### B. 用語集

| 用語 | 定義 |
|------|------|
| 最短価値到達パス | ユーザーが価値を感じるまでの最短ルート |
| 段階的機能開放 | 習熟度に応じて機能を解放する仕組み |
| コンテキストヘルプ | その場で必要な情報を提供するヘルプ |
| プログレッシブディスクロージャー | 必要に応じて情報を段階的に開示 |

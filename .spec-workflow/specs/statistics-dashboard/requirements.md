# Requirements Document: Statistics Dashboard

## Introduction

統計ダッシュボード機能は、コンテスト主催者が応募状況、人気スポット、投票傾向を視覚的に把握できるダッシュボードを提供する。グラフやチャートを活用して、コンテスト運営に必要なデータをリアルタイムで確認でき、より効果的なコンテスト運営判断を支援する。

## Alignment with Product Vision

この機能は product.md に記載された以下の目標に直接貢献する:

- **Monitoring & Visibility**: 「応募状況、投票進捗のリアルタイム表示」「応募数、閲覧数、投票数、エリア別応募分布」の表示
- **Future Vision - Analytics**: 「人気撮影スポット分析、地域別トレンド把握」の実現
- **Business Objectives**: 「運営者の負担を軽減し、継続的なコンテスト開催を支援する」

tech.md で指定された Chartkick + Chart.js を活用してグラフ表示を実装する。

## Requirements

### Requirement 1: 応募数推移グラフ

**User Story:** 主催者として、コンテスト期間中の応募数の推移をグラフで確認したい。これにより応募のピーク時期や停滞期を把握し、プロモーション施策を最適化できる。

#### Acceptance Criteria

1. WHEN 主催者が統計ダッシュボードにアクセス THEN システム SHALL 日別の応募数推移を折れ線グラフで表示する
2. WHEN コンテスト開始前 THEN システム SHALL 「まだデータがありません」メッセージを表示する
3. WHEN 新規応募が登録された THEN システム SHALL グラフデータを更新して最新の応募数を反映する
4. IF 応募期間が7日以上 THEN システム SHALL 週単位でのグループ化オプションを提供する
5. WHEN グラフのデータポイントにホバー THEN システム SHALL その日の正確な応募数をツールチップで表示する

### Requirement 2: 人気スポット分析

**User Story:** 主催者として、どのスポットからの応募が多いかを把握したい。これにより人気エリアの傾向を分析し、次回コンテストの対象エリア選定に活用できる。

#### Acceptance Criteria

1. WHEN 主催者が統計ダッシュボードにアクセス THEN システム SHALL スポット別応募数を棒グラフで表示する（上位10件）
2. WHEN スポット別グラフを表示 THEN システム SHALL 応募数の多い順にソートして表示する
3. IF 応募にスポット情報がない THEN システム SHALL 「スポット未指定」としてカウントする
4. WHEN 棒グラフのバーをクリック THEN システム SHALL そのスポットからの応募一覧へ遷移する
5. WHEN エリアが設定されている THEN システム SHALL エリア別の応募数も円グラフで表示する

### Requirement 3: 投票分析

**User Story:** 主催者として、投票の傾向を把握したい。これにより投票参加状況を監視し、投票促進施策の効果を測定できる。

#### Acceptance Criteria

1. WHEN 主催者が統計ダッシュボードにアクセス THEN システム SHALL 日別の投票数推移を折れ線グラフで表示する
2. WHEN 投票分析を表示 THEN システム SHALL 総投票数、ユニーク投票者数、平均投票数/作品を表示する
3. IF 投票期間が開始前 THEN システム SHALL 「投票期間開始後に表示されます」メッセージを表示する
4. WHEN 投票が行われた THEN システム SHALL グラフデータを更新して最新の投票状況を反映する
5. WHEN 投票分析を表示 THEN システム SHALL 上位得票作品（Top 5）のリストを表示する

### Requirement 4: サマリーカード

**User Story:** 主催者として、コンテストの主要指標を一目で把握したい。詳細なグラフを見る前に全体像を素早く確認できるようにする。

#### Acceptance Criteria

1. WHEN 主催者が統計ダッシュボードにアクセス THEN システム SHALL ページ上部に主要指標のサマリーカードを表示する
2. WHEN サマリーカードを表示 THEN システム SHALL 以下の指標を含める: 総応募数、総投票数、参加ユーザー数、登録スポット数
3. WHEN 前日比データがある THEN システム SHALL 各指標の前日比増減を表示する（+5 件など）
4. IF データが0件 THEN システム SHALL 「0」と表示し、適切なアイコンを使用する

### Requirement 5: アクセス制御

**User Story:** 主催者として、自分のコンテストの統計のみを閲覧したい。他の主催者のデータは見えないようにする。

#### Acceptance Criteria

1. WHEN 未認証ユーザーがダッシュボードにアクセス THEN システム SHALL ログインページにリダイレクトする
2. WHEN 認証済み主催者がダッシュボードにアクセス THEN システム SHALL 自分が主催するコンテストの統計のみを表示する
3. IF 他の主催者のコンテストID指定でアクセス THEN システム SHALL アクセス拒否メッセージを表示しリダイレクトする
4. WHEN 管理者がダッシュボードにアクセス THEN システム SHALL 全コンテストの統計を閲覧可能にする

## Non-Functional Requirements

### Code Architecture and Modularity

- **Single Responsibility Principle**: 統計計算ロジックはサービスクラス（StatisticsService）に分離する
- **Modular Design**: グラフコンポーネントは再利用可能なパーシャルとして実装する
- **Dependency Management**: Chartkick/Chart.js 以外の外部ライブラリ追加を最小限に抑える
- **Clear Interfaces**: サービスクラスは明確なパブリックAPIを持ち、コントローラーから呼び出す

### Performance

- ダッシュボードページの初期表示: 2秒以内
- グラフデータの取得: 1秒以内
- 1000件以上の応募があるコンテストでも快適に動作すること
- 必要に応じてキャッシュを活用し、重い集計クエリを最適化する

### Security

- 主催者は自分のコンテストデータのみ閲覧可能
- 統計データへのAPIアクセスも認証・認可を必須とする
- SQLインジェクション対策としてパラメータ化クエリを使用する

### Reliability

- 統計計算でエラーが発生しても、他のセクションは正常に表示する
- データがない状態でも適切なフォールバック表示を行う

### Usability

- グラフは視覚的に分かりやすい配色を使用する
- モバイルでも閲覧可能なレスポンシブデザイン
- 主要指標は日本語でラベル表示する
- ツールチップやヘルプテキストで各指標の意味を説明する

# Requirements Document

## Introduction

コンテスト作成・管理機能は、運営者がフォトコンテストを作成・編集・管理できる機能を提供する。テーマ設定、応募期間、説明文などの基本情報を設定し、コンテストのライフサイクル（下書き→公開→終了）を管理できる。

## Alignment with Product Vision

この機能は product.md で定義された「コンテスト作成・管理: テーマ設定、応募期間、審査基準の設定機能」を実現する。運営者の負担を軽減し、30分以内でコンテストを開始可能にするという成功指標に貢献する。

---

## Requirements

### Requirement 1: コンテスト一覧表示

**User Story:** 運営者として、自分が作成したコンテストの一覧を確認したい。それにより、現在のコンテスト状況を把握できる。

#### Acceptance Criteria

1. WHEN 運営者がダッシュボードからコンテスト一覧ページにアクセスする THEN システムは自分が作成したコンテストの一覧を表示する
2. WHEN コンテスト一覧を表示する THEN システムはコンテスト名、ステータス、応募期間、応募数を表示する
3. IF コンテストが存在しない THEN システムは「コンテストがありません」というメッセージと新規作成ボタンを表示する
4. WHEN 運営者がコンテスト一覧を表示する THEN システムはステータス（下書き、公開中、終了）でフィルタリングできる

---

### Requirement 2: コンテスト新規作成

**User Story:** 運営者として、新しいフォトコンテストを作成したい。それにより、地域の写真コンテストを開催できる。

#### Acceptance Criteria

1. WHEN 運営者が新規作成ボタンをクリックする THEN システムはコンテスト作成フォームを表示する
2. WHEN 運営者がフォームに必要情報を入力して保存する THEN システムはコンテストを下書き状態で保存する
3. IF 必須項目（タイトル）が未入力 THEN システムはエラーメッセージを表示して保存しない
4. WHEN コンテストが作成される THEN システムは作成したユーザーを所有者として関連付ける

**必須入力項目:**
- タイトル（100文字以内）

**任意入力項目:**
- 説明文（テキスト）
- テーマ（255文字以内）
- 応募開始日時
- 応募終了日時
- サムネイル画像

---

### Requirement 3: コンテスト編集

**User Story:** 運営者として、作成したコンテストの情報を編集したい。それにより、誤りの修正や情報の更新ができる。

#### Acceptance Criteria

1. WHEN 運営者がコンテスト詳細画面で編集ボタンをクリックする THEN システムは編集フォームを表示する
2. WHEN 運営者が編集フォームで変更を保存する THEN システムは更新された情報を保存する
3. IF 他のユーザーが作成したコンテストを編集しようとする THEN システムはアクセスを拒否する
4. WHEN 公開中のコンテストを編集する THEN システムは即座に変更を反映する

---

### Requirement 4: コンテスト詳細表示

**User Story:** 運営者として、コンテストの詳細情報を確認したい。それにより、コンテストの状態や設定を確認できる。

#### Acceptance Criteria

1. WHEN 運営者がコンテスト一覧から特定のコンテストをクリックする THEN システムはコンテストの詳細情報を表示する
2. WHEN コンテスト詳細を表示する THEN システムは全ての設定情報とステータスを表示する
3. WHEN コンテスト詳細を表示する THEN システムは編集・公開・終了などのアクションボタンを表示する

---

### Requirement 5: コンテスト公開

**User Story:** 運営者として、下書きのコンテストを公開したい。それにより、参加者が応募できるようになる。

#### Acceptance Criteria

1. WHEN 運営者が下書きコンテストの公開ボタンをクリックする THEN システムは確認ダイアログを表示する
2. WHEN 運営者が公開を確認する THEN システムはコンテストのステータスを「公開中」に変更する
3. IF 応募開始日時が設定されている AND 現在時刻が応募開始日時より前 THEN システムは「公開予定」として表示する
4. IF タイトルが未設定 THEN システムは公開を拒否してエラーメッセージを表示する

---

### Requirement 6: コンテスト終了

**User Story:** 運営者として、公開中のコンテストを終了したい。それにより、新規応募を締め切ることができる。

#### Acceptance Criteria

1. WHEN 運営者が公開中コンテストの終了ボタンをクリックする THEN システムは確認ダイアログを表示する
2. WHEN 運営者が終了を確認する THEN システムはコンテストのステータスを「終了」に変更する
3. WHEN コンテストが終了する THEN システムは新規応募を受け付けなくなる
4. IF 応募終了日時を過ぎた THEN システムは自動的にコンテストを終了する

---

### Requirement 7: コンテスト削除

**User Story:** 運営者として、不要になったコンテストを削除したい。それにより、一覧を整理できる。

#### Acceptance Criteria

1. WHEN 運営者がコンテストの削除ボタンをクリックする THEN システムは確認ダイアログを表示する
2. WHEN 運営者が削除を確認する THEN システムはコンテストを論理削除する（実際のデータは保持）
3. IF 公開中のコンテストを削除しようとする THEN システムは削除を拒否してエラーメッセージを表示する
4. IF コンテストに応募作品がある THEN システムは警告メッセージを表示する

## Non-Functional Requirements

### Code Architecture and Modularity
- **Single Responsibility Principle**: Contest モデルはコンテスト情報のみを管理し、応募や投票は別モデルで管理
- **Modular Design**: ContestsController は標準的な RESTful パターンに従う
- **Dependency Management**: 既存の Organizers::BaseController を継承して認証・権限を再利用
- **Clear Interfaces**: ステータス管理は enum で明確に定義

### Performance
- コンテスト一覧表示: 1秒以内
- ページネーション: 1ページあたり20件
- 画像アップロード: 5MB以下

### Security
- 運営者のみがアクセス可能（Organizers::BaseController で保証）
- 所有者のみが編集・削除可能
- CSRF対策（Rails標準）

### Reliability
- データ削除は論理削除で実装（deleted_at カラム）
- ステータス変更は履歴を記録

### Usability
- フォームは日本語ラベルを使用
- エラーメッセージは日本語で表示
- 確認ダイアログは重要な操作（公開、終了、削除）で表示

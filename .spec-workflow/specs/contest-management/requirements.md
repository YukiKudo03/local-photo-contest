# Requirements Document

## Introduction

運営組織（コンテスト主催者）向けのアカウント認証システム。自治体、商店街、観光協会などのコンテスト運営者が、安全にシステムにログインし、コンテストを管理できる機能を提供する。Deviseを活用したRails標準の認証機構を基盤とし、運営者専用の名前空間（Organizers::）でアクセス制御を行う。

## Alignment with Product Vision

本機能はproduct.mdの以下の目標を支援する：

- **シンプル運営**: 専門知識がなくても主催・参加できる使いやすさを実現するため、直感的なログイン・認証フローを提供
- **コンテスト作成・管理**: 運営者がコンテストを作成・管理するための認証基盤を確立
- **セキュリティ**: 不正アクセス防止、なりすまし防止のための堅牢な認証システム

ターゲットユーザーである「自治体・観光協会」「商店街・地域団体」「地域メディア」の担当者が、PCおよびモバイルから安全にアクセスできる認証システムを構築する。

## Requirements

### Requirement 1: 運営者アカウント登録

**User Story:** As a 運営者（コンテスト主催者）, I want アカウントを新規登録できる, so that システムにログインしてコンテストを管理できる

#### Acceptance Criteria

1. WHEN 運営者がメールアドレスとパスワードを入力して登録ボタンを押す THEN システム SHALL アカウントを作成し確認メールを送信する
2. IF パスワードが8文字未満である THEN システム SHALL エラーメッセージを表示し登録を拒否する
3. IF 既に登録済みのメールアドレスが入力された THEN システム SHALL 「このメールアドレスは既に使用されています」と表示する
4. WHEN 確認メール内のリンクをクリックする THEN システム SHALL アカウントを有効化しログイン画面にリダイレクトする
5. IF 確認リンクの有効期限（24時間）が切れている THEN システム SHALL 新しい確認メールの再送信オプションを表示する

### Requirement 2: 運営者ログイン

**User Story:** As a 運営者, I want メールアドレスとパスワードでログインできる, so that 自分のコンテスト管理画面にアクセスできる

#### Acceptance Criteria

1. WHEN 正しいメールアドレスとパスワードを入力してログインボタンを押す THEN システム SHALL 運営者ダッシュボードにリダイレクトする
2. IF メールアドレスまたはパスワードが間違っている THEN システム SHALL 「メールアドレスまたはパスワードが正しくありません」と表示する
3. WHEN 「ログイン状態を保持する」にチェックを入れてログインする THEN システム SHALL 2週間セッションを維持する
4. IF アカウントが未確認の状態でログインを試みる THEN システム SHALL 「アカウントの確認が必要です」と表示し確認メール再送信オプションを提供する
5. WHEN ログインに5回連続で失敗する THEN システム SHALL アカウントを30分間ロックする

### Requirement 3: パスワードリセット

**User Story:** As a 運営者, I want パスワードを忘れた場合にリセットできる, so that アカウントへのアクセスを回復できる

#### Acceptance Criteria

1. WHEN 「パスワードを忘れた」リンクをクリックしてメールアドレスを入力する THEN システム SHALL パスワードリセットメールを送信する
2. IF 入力されたメールアドレスが登録されていない THEN システム SHALL セキュリティのため同じ成功メッセージを表示する（アカウント存在の漏洩防止）
3. WHEN リセットメール内のリンクをクリックする THEN システム SHALL 新しいパスワード設定画面を表示する
4. IF リセットリンクの有効期限（6時間）が切れている THEN システム SHALL 「リンクの有効期限が切れています」と表示し再発行オプションを提供する
5. WHEN 新しいパスワードを設定する THEN システム SHALL パスワードを更新し自動的にログインさせる

### Requirement 4: ログアウト

**User Story:** As a 運営者, I want 安全にログアウトできる, so that 共有端末でも安心してシステムを使用できる

#### Acceptance Criteria

1. WHEN ログアウトボタンをクリックする THEN システム SHALL セッションを無効化しログイン画面にリダイレクトする
2. WHEN ログアウトが完了する THEN システム SHALL 「ログアウトしました」と確認メッセージを表示する
3. IF ログアウト後にブラウザの戻るボタンを押す THEN システム SHALL 認証が必要なページへのアクセスを拒否しログイン画面を表示する

### Requirement 5: 運営者専用エリアのアクセス制御

**User Story:** As a システム, I want 未認証ユーザーの運営者エリアへのアクセスを拒否する, so that コンテスト管理機能のセキュリティを確保できる

#### Acceptance Criteria

1. IF 未認証ユーザーが運営者専用URL（/organizers/*）にアクセスする THEN システム SHALL ログイン画面にリダイレクトする
2. WHEN リダイレクト後にログインに成功する THEN システム SHALL 元々アクセスしようとしていたページにリダイレクトする
3. IF 認証済みだが運営者権限を持たないユーザーが運営者エリアにアクセスする THEN システム SHALL 403 Forbiddenエラーページを表示する

### Requirement 6: セッション管理

**User Story:** As a 運営者, I want セッションが適切に管理される, so that セキュリティと利便性のバランスが取れた状態でシステムを使用できる

#### Acceptance Criteria

1. IF セッションが24時間以上経過する AND 「ログイン状態を保持」がオフの場合 THEN システム SHALL 自動的にログアウトする
2. WHEN 新しいデバイス/ブラウザからログインする THEN システム SHALL 既存のセッションを維持したまま新しいセッションを作成する
3. IF ユーザーが30分以上操作しない THEN システム SHALL 次の操作時にセッションの有効性を確認する

### Requirement 7: 利用規約同意確認

**User Story:** As a 運営者, I want 利用規約に同意してからサービスを利用できる, so that 法的に有効な同意のもとでサービスを利用できる

#### Acceptance Criteria

1. WHEN 運営者が初回ログインする THEN システム SHALL 利用規約同意画面を表示し、同意するまでダッシュボードへのアクセスを制限する
2. WHEN 運営者が利用規約に同意する THEN システム SHALL 同意日時、同意時のIPアドレス、同意した規約バージョンをデータベースに記録する
3. IF 利用規約が更新された THEN システム SHALL 次回ログイン時に新しい利用規約への同意を求める
4. WHEN 運営者が更新された利用規約に同意する THEN システム SHALL 新しい同意記録を追加し、過去の同意記録は履歴として保持する
5. IF 運営者が利用規約に同意しない THEN システム SHALL 運営者エリアへのアクセスを許可せずログアウトオプションを表示する
6. WHEN 管理者が利用規約を確認する THEN システム SHALL 各ユーザーの同意履歴（日時、IP、バージョン）を一覧表示できる

## Non-Functional Requirements

### Code Architecture and Modularity
- **Single Responsibility Principle**: 認証ロジックはDeviseとconcernsに集約し、コントローラーは薄く保つ
- **Modular Design**: 認証関連のヘルパーメソッドはApplicationControllerのconcernとして実装
- **Dependency Management**: Devise標準機能を最大限活用し、カスタマイズは最小限に抑える
- **Clear Interfaces**: Organizers名前空間内のコントローラーは共通の認証フィルターを継承

### Performance
- ログイン処理は2秒以内に完了すること
- セッション検証は各リクエストで100ms以内に完了すること
- データベースクエリは認証処理あたり最大3クエリ以内

### Security
- パスワードはbcryptで暗号化（コスト係数12以上）
- CSRF対策はRails標準機能を使用
- セッションIDは認証成功時に再生成（Session Fixation対策）
- ブルートフォース攻撃対策としてアカウントロック機能を実装
- HTTPSのみでの運用を前提とする

### Reliability
- 認証サービスは99.9%の可用性を目標とする
- データベース障害時は適切なエラーメッセージを表示
- メール送信失敗時はリトライ機能を実装

### Usability
- 日本語UIで分かりやすいエラーメッセージを表示
- モバイル対応（レスポンシブデザイン）
- フォーム入力時のリアルタイムバリデーション表示
- パスワード強度インジケーターの表示
